# Отчёт по итоговому проекту по курсу «Инженерия Искусственного Интеллекта»

## 1. Паспорт проекта

- **Название проекта:** `Сравнительный анализ архитектур нейронных сетей для сегментации пневмоторакса`
- **Автор:** `Савельев Роман Сергеевич`
- **Группа:** `БФБО-01-23`
- **Контакт:** `frasermuck@gmail.com`
- **Ссылка на репозиторий:** `https://github.com/CrystalRoman/aie-group.git`

Проект посвящён задаче автоматической сегментации пневмоторакса на рентгеновских снимках грудной клетки.  
В рамках работы были реализованы, обучены и сравнены три архитектуры: `UNet`, `AttentionUNet` и `TransformerUNet`.  
Итоговая цель — получить воспроизводимый ML-пайплайн с обучением, артефактами, сравнением моделей по метрикам и минимальным API для инференса.

## 2. Постановка задачи и контекст

### 2.1. Предметная область и задача

Проект относится к области медицинской визуализации. Требуется автоматически находить область пневмоторакса на рентгеновском снимке грудной клетки и строить её бинарную маску.

Потенциальный пользователь такого решения - исследователь или инженер, который хочет:
- автоматически сегментировать подозрительную область на рентгеновском снимке;
- сравнить несколько архитектур сегментации;
- получить воспроизводимый пайплайн обучения и инференса;
- продемонстрировать работу модели через API.

### 2.2. Формулировка задачи в терминах ML/ИИ

- **Входные данные:** рентгеновское изображение грудной клетки.
- **Выход модели:** бинарная маска области пневмоторакса.
- **Тип задачи:** семантическая сегментация.
- **Ограничения и требования:**
  - корректное восстановление маски на медицинских изображениях;
  - устойчивость к малым и сложным областям поражения;
  - воспроизводимость запуска;
  - возможность показать результат не только в ноутбуке, но и через API.

### 2.3. Целевые метрики качества

Основные метрики:
- `Dice`
- `IoU`

Они выбраны потому, что напрямую оценивают качество совпадения предсказанной и истинной масок и являются стандартными метриками для медицинской сегментации.

## 5. Экспериментальный протокол и результаты

### 5.1. Экспериментальный протокол

- `stage_1_train_images.csv` использовался для построения train/val;
- `stage_1_test_images.csv` использовался как внешний test;
- сравнение моделей проводилось в единой схеме обучения;
- лучшие веса сохранялись в `artifacts/.../models/`;
- история обучения сохранялась в `history_*.csv`;
- итоговый threshold выбирался по артефактам обучения.

### 5.2. Сравнение моделей по метрикам

| Модель | Конфиг | Лучший `val_dice` | Лучший `val_iou` | Комментарий |
|---|---|---:|---:|---|
| UNet | `configs/training_unet.yaml` | 0.4823 | 0.3454 | Baseline |
| AttentionUNet | `configs/training_attention_unet.yaml` | 0.5093 | 0.3738 | Улучшение относительно baseline |
| TransformerUNet | `configs/training_transformer_unet.yaml` | 0.5737 | 0.4298 | Лучшая финальная модель |

### 5.3. Выбор финальной модели

Финальной моделью выбран `TransformerUNet`, так как он показал наилучшие значения `Dice` и `IoU` среди всех протестированных архитектур.

Trade-off:
- трансформерная модель тяжелее baseline;
- зато итоговое качество сегментации оказалось выше;
- для финальной демонстрации она была выбрана как наиболее сильная модель.

При этом для **демонстрационной конфигурации API внутри git-репозитория** используется `UNet`.  
Причина в том, что веса `AttentionUNet` и `TransformerUNet` значительно тяжелее и не включены в репозиторий как обычные git-файлы из-за размера, поэтому:
- хранятся отдельно;
- могут быть показаны отдельно на защите;
- при необходимости могут быть вручную положены в папку `artifacts/.../models/`.

Таким образом:
- **исследовательский итог проекта:** лучшая по метрикам модель — `TransformerUNet`;
- **демонстрационный API в репозитории:** `UNet`, вес которой включён в проект и может быть загружен сразу после клонирования.

Если требуется запуск сервиса с другой моделью, достаточно:
1. положить соответствующий файл `best_*.pt` в нужную папку `artifacts/<имя_эксперимента>/models/`;
2. поменять `training_config_path` в `configs/service.yaml`;
3. перезапустить сервис или вызвать `/reload`.

Гугл диск с моделями: `https://drive.google.com/drive/folders/1B14T_MM_XE12I90SwCTHWcHj6Hu5loau?usp=drive_link`

## 6. Архитектура решения и сервис

### 6.1. Архитектура пайплайна

Пайплайн проекта включает:
1. чтение и фильтрацию данных;
2. построение dataset/dataloader;
3. обучение модели;
4. сохранение чекпоинтов, history и графиков;
5. инференс через FastAPI.

### 6.2. API и endpoints

Реализованы endpoints:
- `GET /health`
- `POST /predict`
- `POST /reload`

`/predict`:
- принимает PNG/JPG изображение;
- прогоняет его через реальную обученную модель;
- строит бинарную маску;
- сохраняет `mask` и `overlay`;
- возвращает JSON с результатом.

Параметры `/predict`:
- `file`
- `threshold` — необязательный параметр бинаризации

Возвращает:
- `status`
- `model_name`
- `threshold`
- `input_filename`
- `image_size`
- `positive_pixels`
- `mask_coverage`
- `mask_path`
- `overlay_path`
- `mask_url`
- `overlay_url`

### 6.3. Технологический стек

Используются:
- Python
- PyTorch
- timm
- FastAPI
- Uvicorn
- PIL / Pillow
- matplotlib
- pytest

Сервис запускается:
- локально через консоль;
- в Datasphere через ноутбук `notebooks/service_api_check.ipynb`.

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1. Логи и наблюдаемость

- логи обучения сохраняются в `artifacts/.../logs/`;
- сервис использует `logging`;
- endpoint `/health` позволяет быстро проверить состояние сервиса.

### 7.2. Конфигурации

Основные параметры вынесены в:
- `configs/training_unet.yaml`
- `configs/training_attention_unet.yaml`
- `configs/training_transformer_unet.yaml`
- `configs/service.yaml`
- `.env.example`

Это позволяет менять:
- пути к данным;
- параметры обучения;
- выбор модели для сервиса;
- threshold и другие настройки без изменения кода.

### 7.3. Безопасность

- реальные секреты в репозиторий не включаются;
- `.env.example` используется как шаблон;
- большие данные и тяжёлые модели не обязательно хранить в git;
- в репозиторий рекомендуется включать только sample-поднабор данных.

## 8. Ограничения и дальнейшая работа

Ограничения:
- финальная схема обучения использует positive-only режим и фильтрацию маленьких масок;
- API реализован как минимальный FastAPI-сервис;
- в Datasphere Swagger UI может быть недоступен из-за ограничений порта;
- запуск сервиса с финальной моделью зависит от наличия веса модели в `artifacts/.../models/`.

Что можно улучшить:
- расширить инференс-пайплайн;
- добавить более строгую оценку на полном наборе;
- улучшить обработку маленьких областей;
- добавить Docker-сценарий или Git LFS для больших весов;
- расширить тестовое покрытие.
